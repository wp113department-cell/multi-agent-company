07 — Tool / MCP Specification: ~55% done
Done and verified:

Tool Access Matrix is real and enforced structurally, not by instruction: backend/app/agents/tools.py defines READ_ONLY_TOOLS, CODER_TOOLS = READ_ONLY_TOOLS + [edit_file, write_file, git_diff, bash, submit_patch], QA_TOOLS, REVIEWER_TOOLS (read-only, no edit/bash), DEVOPS_TOOLS (bash but read-only health checks), RESEARCH_TOOLS (web+GitHub read only, no bash/write) — matches the spec's matrix row-for-row.
No deploy credentials anywhere in any tool list — grep confirms no "deploy" tool/handler exists in tools.py.
PreToolUse-style enforcement is real: guardrails.py (check_path/check_command) denylists .env, secrets/, .github/workflows/, .git/, and blocks rm -rf, git push --force, DROP DATABASE, etc.; called unconditionally in base_graph.py:_policy_check() before every tool execution (execute_tools node, line 566) — matches "enforced at the application layer... regardless of role."
A genuine custom MCP server exists for Repository Intelligence: backend/app/mcp/server.py — real JSON-RPC 2.0 stdio server (initialize, tools/list, tools/call) exposing index_repository, search_symbols, build_context, query_dependencies, semantic_search, get_file_summary. This is the one server the spec explicitly calls for building custom (10_Repository_Intelligence spec).
GitHub write capability is real: backend/app/tools/git_push_tool.py — create_github_pr() calls the real GitHub REST API ({settings.github_api_base_url}/repos/{repo}/pulls), used by push_and_create_pr().
Web search real (DuckDuckGo, no API key) for Research Agent: tools.py:1136-1155.
Missing/deviated:

The spec's central claim — "Tools are standardized through MCP... used directly rather than reimplemented" for Filesystem/Git/GitHub/Postgres/Web-Search MCP servers — is not what's actually built. Grep for "Model Context Protocol", "MCPServer", Postgres-MCP, Filesystem-MCP, Git-MCP integration returns nothing outside the one custom repo-intel server. All filesystem/git/read/edit/bash capability is implemented as bespoke in-process Python functions styled as Anthropic tool-use handlers (make_read_only_handlers, make_coder_handlers, etc. in tools.py), not as connections to standard MCP servers. Functionally equivalent, architecturally a full deviation from the "use existing MCP servers" principle.
No literal "Postgres MCP" (agents don't get scoped DB read access via any such server; DB access, where it exists, is baked into specific agent tool handlers instead).
No public MCP registry search step evidenced anywhere ("Adding a New Tool" checklist step 1).

Migration strategy matches ("single migration tool, versioned, checked-in") — Alembic confirmed (backend/migrations/env.py imports alembic), 17 sequential version files 001-017, each mapping to real schema growth (phase4/5/6 tables, repos, system_settings, agent_benchmarks, prompt_versions, versioned_lessons, pending_approvals, task_images, etc.) — well beyond the spec's Stage 1-6 scope (expected, given later specs 11-19 exist).

Missing/deviated:

dev_tasks table is missing spec's priority, assigned_agent, project, final_summary columns entirely — confirmed by full read of the DevTask class (models.py:56-97); these are faked in the API layer (see 08 findings) rather than real columns. Table instead grew epic_id, repo_id, branch_name, pr_url, pr_status (legitimate but undocumented-by-spec evolution).
Spec's Stage 3 embeddings table (source_type: code_summary|doc|past_task) doesn't exist as such — the actual pgvector table is memory_embeddings (task-outcome-scoped, outcome/category columns), serving Memory System spec 11 rather than generic repo-intel embeddings.
Spec's status CHECK (status IN (...)) constraints are not implemented as DB constraints — status is a free String(50) column; validity is enforced only in application code (VALID_TRANSITIONS dict). A direct SQL write could set an invalid status.
dev_tasks(project) index doesn't exist (consistent with project column not existing).
agent_runs(task_id) index from spec not found in migration 001 (only status/task_logs indexes created).
Repo-Intelligence-specific tables indexed_files, symbols, call_edges exist as real ORM models and are migrated (models.py:200-238, present in migration 001) but have zero real writers anywhere in the codebase — grep for IndexedFile(, Symbol(, CallEdge( instantiations (excluding class defs) returns nothing. A real gap worth flagging distinctly since it looks complete (tables + migration exist) but isn't functionally connected.
10 — Repository Intelligence Specification: ~60% done
Done and verified:

Repository Scanner real, Tree-sitter based: backend/app/repo_tools/scanner.py:1,11-13 imports tree_sitter, tree_sitter_python, tree_sitter_javascript (confirmed in requirements.txt); index_repository() (line 172) walks the filesystem, parses .py/.js files, extracts SymbolInfo (functions/classes with line numbers) and imports. Supports incremental re-parse via known_hashes content-hash comparison (lines 179-209).
Dependency/Import graph real: build_call_graph() in scanner.py:237-262 builds {caller_file: [callee_file,...]} from import statements (file-level, not function-level — see gap below).
Context Builder + Context Cache real and matches spec almost verbatim: backend/app/repo_tools/context_builder.py — build_context() returns {relevant_files, dependency_chain, related_symbols, summary} combining keyword scoring + semantic search + dependency chain (lines 53-131); explicit in-memory cache keyed on SHA-256(task+repo) with invalidate_context_cache() called after reindex (context_builder.py:12-31, called from api/repo.py:341).
Embedding Search real (Voyage AI, pgvector-adjacent): repo_tools/embeddings.py — generate_embeddings()/semantic_search() with cosine similarity, gracefully degrades to keyword scoring when VOYAGE_API_KEY unset (matches mcp/server.py's semantic_search tool description).
MCP Interface real: backend/app/mcp/server.py exposes query_dependencies, search_symbols/semantic_search (renamed from spec's search_semantic but same function) as real JSON-RPC tools, consumed conceptually by Architect via build_context in the shared graph pipeline.
Update/Re-index: incremental reindex via POST /api/repo/reindex (api/repo.py:345-348, background task) is real; weekly full re-index is also real — main.py:72-84 _weekly_reindex_loop(), started as an asyncio.create_task() at main.py:288 in lifespan().
Missing/deviated:

Call Graph is not function-level as the spec explicitly requires ("maps which functions call which other functions, across file boundaries"). scanner.py's build_call_graph() is actually a file-level import graph mislabeled as a call graph. A genuine function-level call graph exists in repo_tools/ast_engine.py:119-165 (build_call_graph(path, function_name), uses Python ast module to find ast.Call nodes inside function bodies) — but it's single-file only, not cross-file, and is exposed as a standalone chat-agent tool (not part of the Context Builder / MCP server pipeline). So the cross-file, function-level call graph the spec describes doesn't exist as a unified component.
Architecture Mapper is missing entirely — no architecture_mapper.py or equivalent; grep for "architecture_mapper"/"ArchitectureMapper" across the whole backend returns nothing. There's no derived "which services/modules does this belong to" layer.
Symbol Graph is real only in-memory (via index_repository()'s per-file symbol list + search_symbols/find_references tool handlers), not persisted/queryable as a graph DB (the symbols/indexed_files tables that would back this exist in the DB schema but are unwired — see 09 findings).
"Re-indexing must complete before an Architect Agent run is allowed to start" — not enforced anywhere. No blocking/gating logic found between reindex completion and architect_node dispatch in pipeline/graph.py; the Architect just calls build_context/index_repository fresh each run, so there's no formal race-condition guard, just implicit freshness from re-walking the filesystem synchronously.
No webhook-triggered reindex on merge to main — only manual API trigger (POST /api/repo/reindex) and the 7-day timer loop. No CI/git-hook integration found.
"Standalone internal service... any future internal tool can query it" — in practice the MCP server (mcp/server.py) is a stdio script, not a running network service other tools could discover/connect to at will; it's not started by main.py or documented as a long-running process — it exists as invocable code, not a deployed standalone service.
Overall Cross-File Observations
The 72-agent/AGENT_CONTRACT/ensure_all_agents_registered() claims in the task prompt are all independently confirmed correct via direct code reads, not just filenames.
The most consistent pattern of "looks done, isn't wired" across these five specs is the Repository-Intelligence DB persistence layer (indexed_files/symbols/call_edges tables — real schema, zero writers) and the API-layer field faking in dev_tasks (priority/project/assigned_agent/final_summary returned as static placeholders, not real columns).
The MCP principle in spec 07 is the largest philosophical gap: the actual system uses custom in-process tool handlers everywhere except the one genuinely-custom Repository Intelligence MCP server, rather than "use existing MCP servers directly" as spec 07 prescribes.
